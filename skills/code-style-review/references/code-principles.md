# Code Principles

General coding specifications for reference, used for code review, code quality assessment, and code optimization.

## Thirty General Coding Rules

1. **组合法**: 将每个函数拆分成多个子函数，每个子函数执行一个可识别的任务。保持方法中的所有操作处于同一抽象级别。这样自然会产生许多小方法，每个方法只有几行代码。
2. **揭示意图的名字**: 方法/函数的命名应该基于其功能，而不是实现方式。读者无需阅读函数体就能理解调用的目的。
3. **将注释替换为清晰的代码**: 如果注释重复了代码的功能，请删除它。如果无法删除注释，请重构代码（例如，提取一个命名良好的函数，重命名一个变量），直到注释变得多余。注释应该用来解释“为什么”，而不是“是什么”。
4. **构造函数清晰度**: 提供工厂函数或构造函数来创建格式良好的实例。预先传递所有必需的参数，这样调用者就不会收到未初始化的对象。
5. **方法的单一责任制**: 每种方法都应该只有一个改变的理由。如果一种方法需要用一段话来解释，那就说明它需要解释的理由太多了。
6. **只说一遍，别重复**: 所有知识或逻辑都应该只存在于一个地方。重复代码会导致多次更新的麻烦——务必将其删除。
7. **行为重于状态**: 首先要确保行为（公共接口）正确。如果内部实现隐藏在简洁的 API 之后，以后总可以进行更改。过早地优化数据布局会将消费者与实现细节耦合起来。
8. **意图揭示型选择器/函数名称**: 函数名称应该根据其代表的概念来命名，而不是根据其使用的算法。includes(item)例如，`a` 比 `b` 更好linearSearchFor(item)。想象一下第二个截然不同的实现——你会给它起同样的名字吗？如果不会，那就应该使用更通用的名称。
9. **针对深度嵌套的保护条款**: 在函数顶部处理边界情况和错误情况，并尽早返回。主逻辑路径应无需缩进即可阅读。
10. **查询方法返回；命令变更**: 将回答问题（返回值，无副作用）的函数与改变状态的函数分开。查询方法名称以逗号is(, ) 为前缀has，can表示布尔值。
11. **解释变量**: 当复杂的表达式难以阅读时，将其结果赋值给一个命名清晰的局部变量。变量名本身就起到了解释作用。
12. **角色暗示名称**: 变量名应根据其扮演的角色而非类型来命名。employees例如，不要使用` employeeList.` 或 ` .`。变量类型可以根据上下文推断。queryqueryString
13. **使用多态性代替条件语句**: 当相同的 if/switch 结构出现在多个地方时，请将其替换为多态对象，每个对象实现一个分支。添加新的 case 相当于添加一个新类，而不是编辑现有代码。
14. **委托，不要继承（优先使用组合）**: 通过将工作传递给协作对象而不是创建子类来共享实现。委托机制保证了两个对象可以独立替换，并避免了深层次的继承关系。
15. **复杂逻辑的方法对象**: 当一个方法变得庞大且共享大量临时变量时，应将所有计算提取到一个单独的类中。将参数和临时变量转换为实例字段，将逻辑放在一个方法中compute()，call()然后使用组合方法进行简化。
16. **执行环绕（资源括号）**: 当两个操作必须始终同时发生时（例如打开/关闭、锁定/解锁、设置/拆卸），请公开一个接受回调的函数。这样，​​调用者就不会忘记第二个操作。
17. **显式初始化**: 在构造函数创建时初始化所有状态。切勿依赖调用者在创建后按正确顺序设置字段。如果存在默认值，请在构造函数中设置它们。
18. **延迟初始化**: 当计算或获取某个值成本很高且可能并不需要时，应将其延迟到首次访问时再进行。将结果缓存到一个字段中，并在后续调用中返回该缓存值。
19. **常量方法/命名常量**: 用命名常量或无参方法替换魔法字面量。MAX_RETRIES它传递的信息比命名常量更多5，并且方法允许子类覆盖其值。
20. **间接变量访问（封装字段）**: 通过 getter/setter 方法而非直接访问实例字段。这样，您就可以在同一个地方添加验证、日志记录、延迟初始化或变更通知等功能。
21. **收藏品访问安全**: 永远不要从 getter 方法返回原始的可变集合。应该返回副本、不可变视图，或者只公开特定领域的添加/删除/枚举方法。
22. **平等与哈希合约**: 如果重写了相等性判断，则也需要重写哈希算法以进行匹配。相等的对象必须生成相同的哈希值。两者都应基于同一组字段。
23. **调解协议**: 当两个对象进行大量协作时，应明确并一致地定义它们之间的消息传递方式。同时，使用清晰易懂的名称，以便第三方能够实现相同的接口。
24. **跨类型作业的双重调度**: 当行为取决于两个对象的类型（而不仅仅是接收器）时，使用双重分发：接收器使用更具体的方法回调参数，并在方法名中包含自己的类型。
25. **可插拔行为优于子类爆炸**: 当多个子类仅在一两个方法上有所不同时，应使用接受策略（回调函数、lambda 表达式或策略对象）的单一类来取代层级结构。子类化仅用于真正不同的行为类型。
26. **参数采集**: 当多个子方法需要对单个结果集合做出贡献时，应将该集合作为参数传递，而不是连接返回值或将状态存储在字段中。
27. **仅有趣的返回值**: 方法只有在调用者需要时才应该返回值。默认情况下不要返回self内部状态——要么返回有意义的值，要么就什么都不返回。让返回值成为有意为之的决策。
28. **可读流程的逆向方法**: 如果按顺序向多个接收者发送消息会降低可读性，请在参数上添加一个便捷方法，使所有调用都通过同一个对象进行。从左到右的可读流程至关重要。
29. **面向开发人员人体工程学的调试打印**: 重写toString` __repr__//`inspect以显示开发人员调试时所需的结构信息。面向用户的显示字符串是另一个需要单独考虑的问题。
30. **逐步采用模式**: 不要试图一次性应用所有规则。先编写代码，发现问题所在，然后再应用能够解决问题的模式。模式是重构的目标，而不是预先设定的规则。边写边清理。
